# 2.11 тАФ Purchase Tracking (`purchase`)

## What This Does & Why

This subproject instruments the `purchase` event тАФ the most important measurement in the entire ecommerce stack. Every upstream event (`view_item` тЖТ `begin_checkout`) exists to model funnel drop-off relative to this endpoint. For Google Ads, purchase conversion data is the signal Smart Bidding uses to allocate budget across campaigns, ad groups, and keywords. Without it, Google Ads is running blind.

The Shopify checkout runs inside a sandboxed iframe on `checkout.shopify.com`. The Storefront Container (`GTM-NMPNZ4TV` via `theme.liquid`) does not fire there тАФ it is permanently blocked by the checkout environment. The solution is a dedicated Checkout Pixel Container (`GTM-MK4XSM4X`) loaded inside Shopify's Custom Pixel, which runs in a lax sandbox that allows script execution via `document.createElement('script')`. GA4 client ID continuity and Google Ads GCLID attribution are both maintained across the sandbox boundary via Shopify's `browser.cookie.get()` API. See ADR-0015, ADR-0016, ADR-0017.

## Prerequisites

- [x] Storefront Container `GTM-NMPNZ4TV` at `v2.6.0` minimum (subprojects 2.1тАУ2.10 complete)
- [x] GA4 property `G-JPWF3JGT5P` тАФ no additional configuration required before this subproject
- [x] Google Ads account `AW-18244478477` тАФ `Purchase` conversion action must be created before GTM tag configuration (platform sequencing bug: creating the action first ensures the Conversion ID + Label fields appear in GTM)
- [x] GA4 тЖТ Google Ads link must be created **after** the conversion action exists (same sequencing bug)
- [x] Shopify Admin тЖТ Settings тЖТ Customer Events access (to create the Custom Pixel)
- [x] `GTM-MK4XSM4X` Checkout Pixel Container created (separate GTM account/container тАФ not the Storefront Container)

## Business Requirement

Track every completed purchase on the Kova Studio store as a GA4 `purchase` key event and a Google Ads conversion action, with accurate transaction revenue, product-level data, and GCLID attribution.

### Full Custom Pixel Code

This code lives in Shopify Admin тЖТ Settings тЖТ Customer Events тЖТ `Kova - Checkout Pixel`. It is plain JavaScript тАФ no imports, no `register()` call. `analytics` and `browser` are sandbox globals.

[Screenshot: Shopify Admin тЖТ Customer Events тЖТ Kova - Checkout Pixel code editor showing full pixel code]

```javascript
(async () => {
  const gaCookieRaw = await browser.cookie.get('_ga');
  const gclAwRaw    = await browser.cookie.get('_gcl_aw');

  // Parse client_id from _ga: "GA1.1.<p1>.<p2>" тЖТ "<p1>.<p2>"
  const gaParts  = gaCookieRaw ? gaCookieRaw.split('.') : [];
  const clientId = gaParts.length >= 4 ? gaParts.slice(2).join('.') : undefined;

  // Seed sandbox document.cookie with _gcl_aw so the GAds conversion tag
  // can read GCLID natively from the sandbox cookie store (ADR-0017).
  if (gclAwRaw) {
    document.cookie = `_gcl_aw=${gclAwRaw}; path=/`;
  }

  // Initialise Checkout Pixel Container (plain JS тАФ no <script> tags in Custom Pixels)
  window.dataLayer = window.dataLayer || [];
  window.dataLayer.push({ 'gtm.start': new Date().getTime(), event: 'gtm.js' });
  var s = document.createElement('script');
  s.src = 'https://www.googletagmanager.com/gtm.js?id=GTM-MK4XSM4X';
  document.head.appendChild(s);

  analytics.subscribe('checkout_completed', (event) => {
    const checkout = event.data.checkout;
    const order    = checkout.order;

    const items = checkout.lineItems.map((item, index) => ({
      item_id:       String(item.variant.id),
      item_group_id: String(item.variant.product.id),
      item_name:     item.title,
      item_category: item.variant.product.type,
      item_variant:  item.variant.title,
      price:         parseFloat(item.variant.price.amount),
      quantity:      item.quantity,
      index:         index + 1
    }));

    window.dataLayer.push({ ecommerce: null });
    window.dataLayer.push({
      event:     'purchase',
      client_id: clientId,
      ecommerce: {
        transaction_id: String(order.id),
        value:    parseFloat(checkout.totalPrice.amount),
        tax:      parseFloat(checkout.totalTax.amount),
        shipping: parseFloat(checkout.shippingLine?.price?.amount ?? 0),
        currency: checkout.totalPrice.currencyCode,
        items:    items
      }
    });
  });
})();
```

## Architectural Decisions

### GA4 Client ID Continuity (ADR-0015)

GTM initialised inside the sandbox iframe generates a new phantom client ID for the sandboxed `window`. If uncorrected, GA4 would attribute the `purchase` event to a one-time user who has never been seen before, breaking session stitching and funnel analysis.

Fix: `browser.cookie.get('_ga')` reads the real `_ga` cookie across the sandbox boundary (Shopify's sandbox-safe API). The cookie value (`GA1.1.<p1>.<p2>`) is parsed to extract `<p1>.<p2>` тАФ the actual client ID. This value is pushed as `client_id` in the dataLayer push, then overrides the auto-generated ID via GTM's "Fields to Set" on the `GA4 - Config` tag.

```javascript
const gaParts  = gaCookieRaw ? gaCookieRaw.split('.') : [];
const clientId = gaParts.length >= 4 ? gaParts.slice(2).join('.') : undefined;
```

### GCLID Attribution in Sandbox (ADR-0017)

The sandbox iframe's `document.cookie` is empty on initialisation тАФ the storefront's `_gcl_aw` cookie is not inherited. The Google Ads conversion tag reads GCLID by scraping `document.cookie` internally; there is no GTM parameter that accepts a GCLID value from the dataLayer.

Fix: read `_gcl_aw` via `browser.cookie.get('_gcl_aw')`, then write the raw value to the sandbox's own `document.cookie` **before GTM loads**:

```javascript
if (gclAwRaw) {
  document.cookie = `_gcl_aw=${gclAwRaw}; path=/`;
}
```

When `GAds - Conversion - Purchase` fires, it queries `document.cookie`, finds the manually planted `_gcl_aw`, and attaches the GCLID to the conversion hit. Non-paid traffic (no `_gcl_aw`) writes nothing тАФ the conversion is recorded without GCLID attribution, which is correct behaviour.

### Dedicated Container (ADR-0016)

Reusing the Storefront Container (`GTM-NMPNZ4TV`) would cause every tag with an "All Pages" or "Initialization - All Pages" trigger to fire when the sandbox iframe initialises: phantom `page_view` from the GA4 Config tag, phantom remarketing hit from `GAds - Remarketing`, Conversion Linker firing in a context where it serves no purpose. The dedicated container eliminates this structurally тАФ no phantom data reaches GA4 or Google Ads, and no blocking triggers accumulate as the Storefront Container grows.

### Deduplication

`checkout_completed` is a Shopify backend-bound event тАФ it fires once per order and does not re-fire on thank-you page refresh. Platform-level deduplication via `transaction_id` is configured in GA4 as a fail-safe. For Google Ads: **Count: Every** (not One) тАФ `transaction_id` handles duplicate-fire dedup at the platform level; Count: Every ensures that multiple distinct purchases from the same ad click are each counted and attributed.

## Common Errors & Fixes

| Error / Symptom | Root Cause | Fix |
|----------------|------------|-----|
| `client_id` is a UUID-style string, not `XXXXXXXXX.XXXXXXXXXX` | GTM's auto-generated sandbox client ID is not being overridden тАФ `_ga` cookie not found or parse failed | Check `browser.cookie.get('_ga')` returns a value; verify "Fields to Set" in GA4 Config tag has `client_id` тЖТ `{{DLV - client_id}}`; confirm the `gaParts.length >= 4` guard succeeds |
| `purchase` event not appearing in sandbox `window.dataLayer` | GTM script failed to load in sandbox (CSP, network error, wrong container ID) | Check DevTools Network tab for `gtm.js?id=GTM-MK4XSM4X` request; confirm container ID matches; check for JS errors in sandbox console context |
| `purchase` event appears but `ecommerce` is `null` or empty | `checkout_completed` event data shape changed, or `event.data.checkout` is undefined | Log `event.data` raw in the pixel to inspect the shape; check `checkout.lineItems` is an array; verify `order.id` exists |
| Google Ads conversion shows "No recent conversions" permanently | `_gcl_aw` not seeded into sandbox cookie before GTM load тАФ conversion tag fires but cannot find GCLID | Confirm `browser.cookie.get('_gcl_aw')` call is **before** the GTM script injection; verify `document.cookie` write executes; test with a known paid-traffic session |
| Phantom `page_view` appearing in GA4 from checkout sandbox URL | `send_page_view` not set to `false` on GA4 Config tag in Checkout Pixel Container | Edit `GA4 - Config` tag in `GTM-MK4XSM4X` тЖТ add `send_page_view: false` тЖТ republish |
| GTM container diagnostics shows "Unsupported tag implementation on Shopify" | GTM flagging that Google Tag is running in a Custom Pixel | Ignore тАФ intended architecture (ADR-0015). The Google & YouTube app cannot run a dedicated checkout-only container or handle client_id and GCLID injection |
| `import { register } from '@shopify/web-pixels-extension'` syntax error | App Extension syntax used instead of Custom Pixel syntax | Remove the import and `register()` wrapper; Custom Pixels use plain JS with `analytics` and `browser` as globals |
| Double-fire: `purchase` appears twice for one order | Pixel code has two `analytics.subscribe('checkout_completed', ...)` calls | Check pixel code has exactly one subscribe call; `checkout_completed` is backend-bound and will not re-fire on reload |
